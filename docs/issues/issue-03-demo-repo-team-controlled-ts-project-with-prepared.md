## What to build

Create or fork a realistic TypeScript repository the team controls, and author the demo pull request: roughly 80 changed files forming a plausible feature drop, one planted reviewable issue for demo step 4, and a prepared follow-up commit ready to push live for demo step 5. This is human judgment work (what makes a convincing PR), not pipeline code, and it unblocks realistic testing for every pipeline slice.

## Acceptance criteria

- [ ] Repository exists and every team member has push access
- [ ] Open PR with ~80 changed files including some noise files (lockfile, generated code) to exercise the appendix
- [ ] Planted issue documented privately for the presenter
- [ ] Follow-up commit prepared on a branch, ready to push mid-demo
- [ ] Repo/PR URL recorded as a comment on this issue
